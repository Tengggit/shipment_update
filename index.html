一、需求說明

在 Google 試算表中記錄新竹貨運的「大包單號」與「貨態」，原本須人工逐筆到新竹官網輸入單號與驗證碼查詢、再手動回填，耗時且易漏。

目標：建立一個由使用者按一個按鈕即自動完成的系統：

  1. 讀取試算表，挑出需要查詢的單號（B 欄）
  2. 自動到新竹官網查詢（驗證碼以 OCR 自動辨識）
  3. 解析最新「作業時間 + 貨物狀態」
  4. 自動回填到試算表 C 欄

資料表欄位：A＝日期、B＝大包單號、C＝大包單號貨態。

二、系統架構與作業流程

本系統採「本機程式 + 瀏覽器介面」設計。使用者在瀏覽器按下按鈕後，由本機 Python（Flask）後端統籌整個流程——先以公開 CSV 讀取 Google 試算表挑出待查單號，逐筆連線新竹官網查詢貨態（圖形驗證碼交由 OCR 自動辨識），解析出最新狀態後，再透過 Apps Script Webhook 寫回試算表 C 欄。

其中「讀取」用公開連結、免授權；「自動寫回」則由 Apps Script Webhook 負責——它是部署在該 Google 試算表上的一段小程式，對外提供一個網址，讓本機程式以密鑰安全呼叫，並依「大包單號」比對到正確的列後寫入，因此免去 Google Cloud 金鑰等繁複設定。

① 使用者啟動在瀏覽器頁面按下「開始查詢並更新」按鈕
↓
② 本機後端（Python / Flask）接手統籌整體流程，並負責驗證碼 OCR 與結果解析
↓
③ 讀取 Google 試算表（公開 CSV）取得 A／B／C 欄，篩選出「C 欄空白或未送達」的大包單號
↓
④ 查詢新竹官網（逐筆）取頁面 → 抓 4 碼驗證碼 → OCR 辨識（猜錯自動換圖重試）→ 送出查詢
↓
⑤ 解析貨況跟隨加密中轉表單，取出最新「作業時間 + 貨物狀態（營業所）」
↓
⑥ Apps Script Webhook 回填以密鑰呼叫部署於試算表的接口，依大包單號比對列、寫入 C 欄
↓
⑦ 完成並回報每筆結果即時顯示於網頁，結束後輸出統計（查到／查無／失敗）

三、技術調查與關鍵發現

  • 新竹查詢頁為 ASP.NET 表單，需帶 __VIEWSTATE 與 Session Cookie；一組 4 碼純數字圖形驗證碼。
  • 查詢成功後會回傳一個自動送出的加密中轉表單（no/chk），再導向真正的貨況結果頁。
  • 結果頁為表格結構，最新一筆在最上：作業時間 + 貨物狀態 + 負責營業所，組合為例如 2026/05/05 10:37 順利送達（三民）。
  • 查無資料時頁面顯示紅字「請檢查您的查貨資料」→ 記為「查無資料」。
  • 驗證碼正確性由新竹伺服器把關：OCR 猜錯只會被退回、換一張重試 → OCR 準確率只影響速度、不影響對錯。
  • 多筆同時查詢會觸發官網錯誤頁 → 改採一次一筆，穩定可靠。
  • OCR 採用 ddddocr（專為驗證碼設計），實測幾乎一次命中。

四、查詢／略過規則

對每一列，當 B 欄為 10 碼數字單號，且符合下列任一條件即納入查詢：

  1. C 欄為空白，或
  2. C 欄不包含任何「已送達關鍵字」。

已送達關鍵字（可於 config.json 調整）：順利送達、ARRIVED（英文不分大小寫）。

→ C 欄已含上述關鍵字的列會被略過不查，避免重複查詢已完成的件。

七、操作方式

  1. 雙擊 啟動查詢.bat → 自動開啟瀏覽器至 http://127.0.0.1:5000
  2. 按下「開始查詢並更新」按鈕
  3. 畫面即時顯示每筆查詢與回填狀態，完成後顯示總結

模式切換（config.json）

  • dry_run: true　＝ 只查詢、不寫回（試跑、安全）
  • dry_run: false　＝ 查詢並自動寫回 C 欄（正式，目前設定）
